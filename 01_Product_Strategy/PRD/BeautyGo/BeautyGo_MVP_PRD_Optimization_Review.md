# BeautyGo MVP PRD 优化清单

基于 原始 BeautyGo PRD 的完整阅读，并结合 `tuling-pm-knowledge` skill 中的 PRD、需求分析与埋点规范整理。

本文档目标不是重写整份 PRD，而是把现有版本从“商业提案 + 产品全景方案”优化为“可进入需求评审和开发排期的 MVP 评审版清单”。

## 1. 总体结论

### 1.1 现有 PRD 的优势

- 商业机会判断清晰，明确指出了“上门美妆 C2C 平台”的市场空白。
- 用户画像、冷启动策略、竞品差异化、技术选型、成本估算都较完整。
- 已形成从 MVP 到成熟期的阶段化路线，对外沟通和融资表达较强。
- 已初步具备北极星指标、用户故事、合规关注点等基础元素。

### 1.2 现有 PRD 的主要问题

- 文档更像“创业提案”而不是“研发可执行 PRD”，商业叙事占比过高。
- MVP 边界仍偏宽，部分 P1/P2 能力混入核心链路，容易导致范围漂移。
- 正常流有描述，但异常流、超时流、失败流、人工介入流不完整。
- 核心业务规则未收口，例如取消、退款、爽约、地址可见、联系方式暴露、纠纷处理。
- 页面级说明不足，多个核心流程仍是占位符。
- 数据埋点、指标口径、看板关系未打通，难以支撑 MVP 验证。
- 用户故事的验收标准不一致，尚不足以直接转测。

### 1.3 本轮优化目标

- 将 PRD 主体聚焦到 0-3 个月 MVP。
- 补齐业务规则、状态机、角色权限、埋点、验收标准。
- 把“市场、融资、扩张、生态”降级为附录，避免干扰首版决策。
- 将首版目标压缩到一个核心验证场景：旅拍妆。

## 2. 建议的 PRD 主结构

建议把正式 PRD 重构为以下主线：

1. 产品背景与 MVP 目标
2. MVP 范围与优先级
3. 角色定义与核心场景
4. 核心业务流程
5. 功能需求明细
6. 业务规则与状态流转
7. 数据与埋点
8. 非功能需求与合规要求
9. 验收标准
10. 附录

### 2.1 建议移出主 PRD 的内容

以下内容保留，但建议挪到附录或单独商业材料：

- 市场规模详细测算
- 融资建议
- 10-18 个月扩张期详细规划
- 18 个月后的生态能力
- 品牌合作、SaaS、跨境、学院等长期业务设想

### 2.2 建议保留但压缩的内容

- 竞品分析：压缩成“为什么做 MVP、为什么先做旅拍妆”的输入。
- 用户画像：保留核心角色，不保留过多延展群体。
- 冷启动策略：保留与 MVP 成功相关的最小动作。

## 3. MVP 目标与范围重定义

## 3.1 MVP 核心目标

MVP 只验证三件事：

- 用户愿意在线完成一次上门美妆的完整预约和支付。
- 化妆师愿意通过平台接单、履约并获得收入。
- 双方在服务后愿意留下评价，形成信任闭环。

### 3.2 MVP 核心假设

| 编号 | 假设 | 验证方式 |
| --- | --- | --- |
| H1 | 用户愿意为旅拍妆场景在线预约并支付 | 观察下单率与支付成功率 |
| H2 | 化妆师愿意完成认证并持续接单 | 观察入驻数、接单率、30 日留存 |
| H3 | 服务完成后双方愿意留评价 | 观察评价率与评分分布 |

### 3.3 MVP 场景收敛

建议首版只服务一个主场景：`旅拍妆`。

收敛原因：

- 场景目标明确，妆容需求相对稳定。
- 需求发生在异地/旅游/拍摄节点，上门价值更强。
- 容易与旅拍摄影工作室形成冷启动合作。
- 相比商务妆、婚礼妆，服务标准更容易统一。

### 3.4 MVP 优先级重排

#### P0：必须上线

| 模块 | 功能 | 保留原因 |
| --- | --- | --- |
| 账户体系 | 手机号注册登录、微信授权登录 | 没有身份体系就无法完成交易闭环 |
| 客户侧 | 化妆师列表页 | 验证用户是否愿意浏览并筛选供给 |
| 客户侧 | 化妆师详情页 | 验证作品集、价格、评分是否影响转化 |
| 客户侧 | 提交预约单 | 形成核心交易入口 |
| 客户侧 | 在线支付 | 验证真实支付意愿 |
| 客户侧 | 订单详情与状态查询 | 保证交易可追踪 |
| 客户侧 | 服务后评价 | 建立信任闭环 |
| 化妆师侧 | 入驻认证 | 建立供给准入和平台可信度 |
| 化妆师侧 | 服务项目与价格配置 | 保证供给可被检索和购买 |
| 化妆师侧 | 档期管理 | 决定订单是否可承接 |
| 化妆师侧 | 新订单接受/拒绝 | 形成供给履约闭环 |
| 化妆师侧 | 服务执行与完成确认 | 驱动订单进入完单与结算 |
| 后台 | 化妆师审核 | 保证供给质量 |
| 后台 | 订单监控与投诉处理 | 处理异常订单和早期人工兜底 |

#### P1：可延期

- 即时通讯
- 紧急联系人通知
- 化妆师收入看板
- 历史订单经营分析

#### P2：移出 MVP 主体

- 用户派单/化妆师抢单
- 社区内容
- 在线化妆指导
- 会员体系
- 优惠券与复杂营销活动
- AI 妆容推荐
- 品牌合作与带货

## 4. 角色定义与权限规则

## 4.1 角色定义

| 角色 | 定义 | 核心目标 |
| --- | --- | --- |
| 客户 | 有上门旅拍妆需求的用户 | 快速找到可信赖化妆师并完成服务 |
| 化妆师 | 申请入驻并通过审核的供给方 | 稳定接单、准时履约、获得收入 |
| 平台审核/运营 | 平台内部管理角色 | 审核供给、处理异常、维护交易秩序 |

### 4.2 关键权限表

| 对象 | 客户可见 | 化妆师可见 | 平台可见 |
| --- | --- | --- | --- |
| 化妆师昵称、头像、作品集、评分、起步价 | 是 | 自己 | 是 |
| 客户完整地址 | 下单后自己可见 | 仅接单后可见 | 是 |
| 客户手机号 | 自己可见 | 否，默认脱敏 | 是 |
| 化妆师实名信息 | 否 | 自己 | 是 |
| 订单状态 | 自己订单可见 | 自己订单可见 | 是 |
| 投诉记录 | 自己发起部分可见 | 自己关联部分可见 | 是 |

### 4.3 权限优化建议

- 在化妆师接单前，客户详细地址不得展示给化妆师。
- 在订单建立前，不开放双方直接交换联系方式。
- 未通过审核的化妆师不得出现在客户列表页。
- 被平台冻结或下线的化妆师不可接新单，但保留历史订单查询能力。

## 5. 核心业务流程优化

## 5.1 客户下单流程

1. 用户进入小程序并完成注册登录。
2. 进入化妆师列表页，选择城市、服务日期、价格区间进行筛选。
3. 进入化妆师详情页，查看作品集、评分、服务项目、可预约时间。
4. 点击预约，选择服务项目、预约日期时间、服务地址、备注信息。
5. 提交订单并完成在线支付。
6. 订单进入“待化妆师确认”状态，平台向化妆师推送新订单提醒。
7. 化妆师确认接单后，订单进入“待服务”。
8. 到达预约时间，化妆师开始服务，订单进入“服务中”。
9. 服务完成后，化妆师提交完成，等待客户确认。
10. 客户确认后订单完成，进入评价流程。
11. 客户提交星级与文字评价，平台释放结算流程。

### 5.2 化妆师接单流程

1. 化妆师完成实名认证、资料填写、作品集上传、服务配置。
2. 平台审核通过后，化妆师可以设置可接单档期与服务半径。
3. 收到新订单提醒后，在时限内选择接受或拒绝。
4. 接受后查看客户脱敏信息与完整服务地址。
5. 到达服务时间后点击开始服务。
6. 服务结束后点击完成服务，提交完成确认。
7. 等待客户确认，超时则进入自动确认规则。
8. 订单结算完成后，收入进入待提现或已结算状态。

### 5.3 需要补齐的异常流程

- 支付成功但订单状态未回写
- 化妆师超时未响应
- 化妆师拒单
- 客户支付后主动取消
- 服务前化妆师爽约
- 客户迟到或无法联系
- 服务完成但客户拒绝确认
- 用户发起投诉或退款
- 平台人工介入纠纷

## 6. 订单状态机与规则补齐

## 6.1 订单状态机建议

| 状态 | 触发条件 | 允许操作 | 备注 |
| --- | --- | --- | --- |
| 待支付 | 用户创建订单但未支付 | 支付、取消 | 超时自动关闭 |
| 待化妆师确认 | 支付成功 | 化妆师接受/拒绝 | 需设置确认时限 |
| 待服务 | 化妆师接受 | 客户查看订单、化妆师查看地址 | 到预约时点可开始服务 |
| 服务中 | 化妆师点击开始服务 | 提交完成、发起异常上报 | 记录开始时间 |
| 待客户确认完成 | 化妆师点击完成服务 | 客户确认完成、发起投诉 | 超时自动确认 |
| 已完成 | 客户确认或系统自动确认 | 评价、结算 | 完成交易闭环 |
| 已取消 | 支付前取消、接单前取消等 | 查看订单 | 需记录取消原因 |
| 退款中 | 已支付后发起退款 | 等待平台处理 | 与投诉可关联 |
| 已退款 | 退款完成 | 查看订单 | 记录退款金额 |
| 纠纷中 | 双方争议、平台介入 | 平台处理 | 资金冻结 |

### 6.2 超时规则建议

- 待支付：30 分钟未支付自动关闭。
- 待化妆师确认：30 分钟未响应自动取消并原路退款。
- 待客户确认完成：48 小时未确认自动完单并触发结算。

### 6.3 通知动作建议

- 支付成功：通知客户和化妆师。
- 化妆师接单/拒单：通知客户。
- 订单即将开始：服务前 2 小时提醒双方。
- 服务完成待确认：提醒客户确认。
- 退款/投诉状态变更：提醒相关方和平台运营。

## 7. 取消、退款、纠纷规则优化

## 7.1 取消规则建议

| 场景 | 建议处理 |
| --- | --- |
| 客户在支付前取消 | 直接取消，无费用 |
| 客户支付后、化妆师未接单前取消 | 直接取消，全额退款 |
| 化妆师拒单 | 自动取消，全额退款 |
| 化妆师超时未响应 | 自动取消，全额退款 |
| 客户在化妆师接单后取消 | MVP 先进入人工审核，避免过早设计复杂违约金 |
| 化妆师接单后爽约 | 平台记录违约，全额退款给客户，计入供给处罚 |

### 7.2 退款与纠纷建议

- MVP 阶段不建议设计复杂自动赔付规则，统一采用“平台工单 + 人工判责”。
- 退款原因需标准化枚举，至少包括：
  用户主动取消、化妆师拒单、化妆师超时、化妆师爽约、服务不满意、其他。
- 发生纠纷时，订单转为“纠纷中”，结算冻结。
- 聊天记录、订单状态日志、操作时间戳作为基础仲裁证据。

### 7.3 平台外交易治理建议

- 订单未建立前，不展示电话、微信号、完整地址。
- IM 若后续上线，需要增加联系方式识别与风控提醒。
- 违规引导线下交易可分三级处罚：提醒、限制接单、封禁。
- 需定义申诉入口与平台复核时限。

## 8. 关键业务规则补齐

## 8.1 化妆品归属规则

MVP 建议只保留一条默认规则：

- 默认由化妆师自备化妆品，服务价格已包含基础耗材成本。

本轮原因：

- 降低下单页复杂度。
- 降低客户与化妆师沟通成本。
- 避免在 MVP 阶段引入“客户自带/混合模式”导致履约争议。

### 8.2 服务范围规则

- 化妆师需设置常驻区域与服务半径。
- 列表页只返回可覆盖该服务地址的化妆师。
- 若用户尚未填写地址，则先按城市和商圈粗筛。

### 8.3 档期规则

- 档期由化妆师主动维护。
- 已被锁定的订单时间段不可再次售卖。
- 平台需记录化妆师拒单率和超时率，作为后续排序降权依据。

## 9. 页面级需求补齐建议

## 9.1 客户侧页面

### 页面 A：首页/化妆师列表页

- 页面目标：帮助用户快速发现可预约供给。
- 核心字段：头像、昵称、风格标签、评分、起步价、可预约时间。
- 关键操作：筛选、排序、进入详情页。
- 空态：当前筛选条件下暂无可预约化妆师。
- 错误态：网络异常、定位失败、筛选失败。
- 权限：未登录可浏览，点击预约时要求登录。

### 页面 B：化妆师详情页

- 页面目标：帮助用户判断是否下单。
- 核心字段：作品集、简介、评分、服务项目、价格、服务范围、可预约档期。
- 关键操作：查看作品、选择服务、进入下单。
- 异常：作品集为空时不允许通过审核上线。

### 页面 C：下单页

- 页面目标：完成服务选择与订单创建。
- 核心字段：服务项目、时间、地址、备注、价格明细。
- 关键操作：提交订单、支付。
- 异常：所选时间被抢占、地址超出服务范围、支付失败。

### 页面 D：订单详情页

- 页面目标：承接订单状态变化与用户操作。
- 核心字段：订单状态、服务信息、化妆师信息、地址、时间、价格、售后入口。
- 关键操作：取消订单、查看进度、确认完成、发起投诉。

### 页面 E：评价页

- 页面目标：沉淀信任资产。
- 核心字段：星级、文字、可选标签。
- 关键操作：提交评价。
- 异常：重复评价禁止提交。

## 9.2 化妆师侧页面

### 页面 F：认证申请页

- 页面目标：完成供给准入。
- 核心字段：实名信息、证件上传、作品集、个人简介。
- 关键操作：提交审核。
- 异常：资料不全、实名认证失败、内容审核失败。

### 页面 G：服务配置页

- 页面目标：配置可售卖服务。
- 核心字段：服务名称、时长、价格、描述、风格标签。
- 关键操作：新增、编辑、下架。

### 页面 H：档期管理页

- 页面目标：控制接单可用性。
- 核心字段：日期、时间段、休息时间、预约状态。
- 关键操作：开关档期、批量设置。

### 页面 I：新订单处理页

- 页面目标：在有限时效内确认接单。
- 核心字段：客户需求摘要、服务时间、地址范围、价格。
- 关键操作：接受、拒绝。
- 异常：订单超时自动失效。

### 页面 J：订单执行页

- 页面目标：驱动履约与完单。
- 核心字段：订单状态、客户信息、地址、服务开始/结束按钮。
- 关键操作：开始服务、完成服务、上报异常。

## 10. 第三方能力的失败分支与兜底

| 能力 | 正常用途 | 失败分支 | 兜底建议 |
| --- | --- | --- | --- |
| 地图服务 | 地址解析、距离过滤 | 地址解析失败 | 允许用户手动修正并提示检查 |
| 支付 | 完成交易 | 支付成功但回调失败 | 通过主动补单任务修正状态 |
| 实名认证 | 化妆师准入 | OCR/人脸失败 | 允许重新提交并进入人工复核 |
| IM | 订单内沟通 | 消息发送失败 | MVP 可先不做，改为订单备注 + 平台客服兜底 |

## 11. 数据埋点与指标体系优化

## 11.1 MVP 最小事件集

| 事件名 | 事件描述 | 触发时机 | 关键属性 | 优先级 | 分析目标 |
| --- | --- | --- | --- | --- | --- |
| account_register_success | 注册成功 | 用户完成注册时 | user_id, channel, city | P0 | 注册转化率 |
| artist_list_view | 浏览化妆师列表 | 列表页曝光时 | user_id, city, filters | P0 | 列表访问量 |
| artist_detail_view | 查看化妆师详情 | 详情页进入时 | user_id, artist_id, source | P0 | 详情转化 |
| booking_initiate | 发起预约 | 点击预约时 | user_id, artist_id, service_id | P0 | 下单意愿 |
| order_submit | 提交订单 | 订单创建成功时 | order_id, user_id, artist_id, amount | P0 | 下单转化 |
| payment_success | 支付成功 | 支付回调成功时 | order_id, amount, pay_channel | P0 | 支付转化 |
| artist_accept_order | 化妆师接单 | 接受订单时 | order_id, artist_id, response_time | P0 | 接单率 |
| service_start | 开始服务 | 化妆师开始服务时 | order_id, artist_id, start_time | P1 | 履约时效 |
| order_complete | 订单完成 | 客户确认或系统自动确认时 | order_id, complete_type | P0 | 完单率 |
| review_submit | 提交评价 | 评价成功时 | order_id, rating, has_text | P0 | 评价率 |
| refund_initiate | 发起退款/投诉 | 用户发起售后时 | order_id, reason_code | P0 | 退款与投诉监控 |

### 11.2 指标映射

| 目标 | 指标 | 对应事件 |
| --- | --- | --- |
| 验证需求存在 | 首单完成率 | account_register_success -> order_complete |
| 验证供给可持续 | 化妆师入驻数、接单率、30 日留存 | artist_accept_order 等 |
| 验证服务闭环 | 订单完成率 | payment_success -> order_complete |
| 验证信任建立 | 评价率、平均评分 | review_submit |

### 11.3 核心漏斗

访问 -> 注册 -> 浏览详情 -> 发起预约 -> 提交订单 -> 支付成功 -> 化妆师接单 -> 完成订单 -> 提交评价

### 11.4 数据校验要求

- 上线后 48 小时内完成埋点校验。
- 对比前端触发量、服务端订单量、支付回调量，确认口径一致。
- 校验异常包括漏报、重复上报、属性缺失、时间戳异常。

## 12. 非功能与合规优化建议

## 12.1 非功能要求

建议在正式 PRD 中保留少量可执行项：

- 列表页首屏结果返回时间不超过 3 秒。
- 支付成功后订单状态回写在 10 秒内完成。
- 新订单提醒延迟不超过 30 秒。
- 敏感信息数据库加密存储。
- 全站 HTTPS。

### 12.2 合规建议

- 隐私政策、用户协议、化妆师入驻协议必须在 MVP 上线前准备完成。
- 明确平台与化妆师是合作关系，而非劳动关系。
- 对化妆品安全要求先通过协议约束和抽查实现，不在 MVP 引入复杂质检流程。

## 13. 验收标准重写建议

现有用户故事可以保留，但建议改为“功能验收表”。

## 14. 关键问题清单

以下问题若不在正式 PRD 中补齐，会直接影响研发评审：

1. 化妆师接单超时时间是多少。
2. 已支付订单在何种条件下允许客户直接取消。
3. 客户与化妆师是否允许在接单前互发消息。
4. 完单确认的自动超时时间是否统一为 48 小时。
5. 服务开始/结束是否强依赖 GPS 打卡。
6. 退款与投诉是否共用同一工单流。
7. 后台审核不通过时，化妆师是否可修改后重新提交。
8. 排序逻辑是否优先按距离、评分、价格还是平台推荐。

## 15. 推荐的后续落地顺序

1. 先基于本清单重构正式 PRD 章节。
2. 再补订单状态机图、角色权限表、异常规则表。
3. 然后补页面级需求和 P0 验收表。
4. 最后补埋点表和数据口径，进入技术评审。

## 16. 结论

BeautyGo 现有 PRD 的商业价值表达已经足够强，真正需要补的是“首版如何清晰做出来”。  
建议后续所有评审都围绕一句话展开：

`BeautyGo MVP 只做一件事：验证旅拍妆场景下，上门美妆交易闭环是否成立。`

只要文档始终围绕这个目标展开，范围、规则、埋点和验收就会自然收口，研发和运营也更容易对齐。
